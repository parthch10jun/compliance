# Control Testing & Evidence Workflow

## Overview
This document explains how controls, tests, evidence, and the review/approval workflow are connected in the Compliance Instance application.

## 1. Control Hierarchy

```
Program (e.g., RBI Cyber Security Framework)
  ├── Requirements (Continuous-state compliance)
  │   └── REQ-001: Implement Multi-Factor Authentication
  └── Obligations (Time-bound deliverables)
      └── OBL-002: Quarterly Cyber Security Status Report
          ↓
      Controls (Implementation of requirements/obligations)
          └── CTRL-001: Multi-Factor Authentication (MFA)
              ├── Tests (Verify control effectiveness)
              │   ├── TST-001: MFA Enforcement Verification
              │   └── TST-002: MFA Bypass Attempt Test
              └── Evidence (Proof of control operation)
                  └── EVD-001: MFA Configuration Report
```

## 2. Control Metadata

Each control contains:

### Basic Information
- **Code**: Unique identifier (e.g., CTRL-001)
- **Name**: Descriptive name
- **Description**: Detailed explanation
- **Category**: Access Control, Cryptography, Monitoring, etc.
- **Type**: Preventive, Detective, or Corrective
- **Automation Level**: Manual, Semi-Automated, or Fully Automated

### Operational Details
- **Frequency**: How often the control operates (Continuous, Daily, Weekly, Monthly, Quarterly, Annual)
- **Owner**: Person responsible for the control
- **Department**: Organizational unit
- **Effectiveness**: Effective, Partially Effective, Ineffective, Not Tested

### Relationships (Many-to-Many)
- **linkedRequirementIds**: Requirements this control satisfies
- **linkedObligationIds**: Obligations this control fulfills
- **linkedProgramIds**: Programs this control belongs to
- **linkedEvidenceIds**: Evidence documents for this control
- **linkedRiskIds**: Risks this control mitigates

### Compliance Metrics
- **complianceStatus**: Compliant, Partially Compliant, Non-Compliant
- **complianceScore**: 0-100 percentage
- **testCount**: Total number of tests
- **testsPassed**: Number of passed tests
- **testsFailed**: Number of failed tests
- **lastTestDate**: When last tested
- **nextTestDate**: When next test is due

## 3. Test Structure

Each test (ControlTest) contains:

### Test Definition
- **code**: Unique test identifier (e.g., TST-001)
- **name**: Test name
- **description**: What the test verifies
- **type**: Design, Operating Effectiveness, Walkthrough, Automated, or Manual
- **procedure**: Step-by-step test instructions
- **expectedResult**: What should happen if control is effective

### Test Execution
- **controlId**: Which control this test verifies
- **linkedEvidenceIds**: Evidence used/generated by this test
- **status**: Passed, Failed, Pending, In Progress, Not Started
- **result**: Pass, Fail, or Partial
- **actualResult**: What actually happened during the test
- **findings**: Issues discovered during testing

### Test Scheduling
- **frequency**: How often to run (Quarterly, Monthly, etc.)
- **scheduledDate**: When test is scheduled
- **executedDate**: When test was actually run
- **dueDate**: Deadline for test completion
- **tester**: Person who executes the test
- **reviewer**: Person who reviews test results

## 4. Evidence Structure

Each evidence document contains:

### Document Information
- **code**: Unique evidence identifier (e.g., EVD-001)
- **name**: Evidence name
- **description**: What the evidence proves
- **type**: Report, Screenshot, Certificate, Policy, Log File, etc.
- **fileName**: Actual file name
- **fileSize**: Size in bytes
- **fileType**: MIME type
- **fileUrl**: Location of the file

### Version Control
- **version**: Current version (e.g., "2.0")
- **versionHistory**: Array of all versions with:
  - version number
  - fileName
  - fileUrl
  - fileSize
  - uploadedBy
  - uploadedAt
  - changes description
  - status (Current or Superseded)

### Relationships
- **linkedControlIds**: Controls this evidence supports
- **linkedRequirementIds**: Requirements this evidence proves
- **linkedObligationIds**: Obligations this evidence fulfills
- **linkedTestIds**: Tests that use this evidence

### Approval Workflow
- **status**: Approved, Pending Review, Rejected, Expired, Draft
- **validationStatus**: Sufficient, Insufficient, Needs Update, Not Reviewed
- **uploadedBy**: Who uploaded the evidence
- **uploadedAt**: When uploaded
- **reviewedBy**: Who reviewed it
- **reviewedAt**: When reviewed
- **approvedBy**: Who approved it
- **approvedAt**: When approved
- **rejectionReason**: Why rejected (if applicable)

### Expiration Management
- **expiresAt**: When evidence expires
- **reminderDays**: Days before expiration to send reminder

## 5. Complete Workflow

### Step 1: Control Definition
1. Create a control (e.g., CTRL-001: MFA)
2. Link to requirements/obligations
3. Assign owner and department
4. Set frequency and automation level

### Step 2: Test Planning
1. Define tests for the control (e.g., TST-001: MFA Verification)
2. Write test procedure with clear steps
3. Define expected results
4. Set test frequency and schedule
5. Assign tester and reviewer

### Step 3: Test Execution
1. Tester receives scheduled test notification
2. Tester follows the procedure step-by-step
3. Tester records actual results
4. Tester uploads evidence during/after test
5. Tester marks test as Pass/Fail/Partial
6. Tester documents any findings

### Step 4: Evidence Upload
1. Evidence is uploaded with metadata:
   - Type (Report, Screenshot, etc.)
   - Description
   - Links to control/test/requirement
2. System records:
   - Who uploaded
   - When uploaded
   - File details
3. Evidence status set to "Pending Review"

### Step 5: Evidence Review
1. Reviewer receives notification
2. Reviewer examines the evidence
3. Reviewer makes decision:
   - **Approve**: Evidence is sufficient
   - **Reject**: Evidence is insufficient
4. Reviewer sets validation status:
   - Sufficient
   - Insufficient
   - Needs Update
5. Reviewer adds comments
6. System records review details

### Step 6: Evidence Approval
1. If approved:
   - Status changes to "Approved"
   - Evidence is linked to control/test
   - Expiration date is set
   - Reminder is scheduled
2. If rejected:
   - Status changes to "Rejected"
   - Rejection reason is recorded
   - Uploader is notified to resubmit

### Step 7: Compliance Calculation
1. System calculates control effectiveness:
   - Tests passed / Total tests = Test pass rate
   - Evidence approval status
   - Last test date vs. frequency
2. Control compliance score is updated
3. Requirement/Obligation compliance is recalculated
4. Program compliance score is updated

## 6. How to Know What Evidence and Tests Are Needed

### For a Control:
1. **View Control Detail Page** (`/controls/[id]`)
2. **Tests Section** shows:
   - All tests defined for this control
   - Test status (Passed/Failed/Pending)
   - Test schedule and frequency
   - Test results and findings
3. **Evidence Section** shows:
   - All evidence linked to this control
   - Evidence status (Approved/Pending/Rejected)
   - Evidence validation status
   - Expiration dates

### Evidence Requirements:
- Evidence is determined by:
  1. **Test Procedure**: What proof is needed to verify the test
  2. **Regulatory Requirements**: What documentation regulators require
  3. **Control Type**: Different controls need different evidence types
  4. **Frequency**: How often evidence must be refreshed

### Example: CTRL-001 (MFA)
- **Tests Needed**:
  - TST-001: Verify MFA is enabled for all users
  - TST-002: Attempt to bypass MFA (negative test)
- **Evidence Needed**:
  - EVD-001: MFA Configuration Report showing 100% coverage
  - Screenshots of MFA prompts
  - Access logs showing MFA challenges
  - Exception reports (if any)
- **Frequency**: Quarterly
- **Expiration**: 3 months after upload

## 7. Viewing the Complete Workflow

### On Control Detail Page:
1. **Control Information** - Basic details, owner, effectiveness
2. **Linked Requirements/Obligations** - What this control satisfies
3. **Tests** - All tests with:
   - Test procedure
   - Execution status
   - Results and findings
   - Tester and reviewer
   - Schedule and dates
4. **Evidence** - All evidence with:
   - Document details
   - Approval status
   - Review history
   - Expiration dates
   - Version history
5. **Mitigated Risks** - Risks this control addresses

### Workflow Visibility:
- **Green badges**: Approved evidence, Passed tests
- **Yellow badges**: Pending review, In Progress tests
- **Red badges**: Rejected evidence, Failed tests
- **Gray badges**: Not yet started, Expired

## 8. Review and Approval Mechanism

### Evidence Review Modal
Located in: `src/components/ReviewEvidenceModal.tsx`

**Features**:
- View evidence details
- Approve or Reject decision
- Set validation status (Sufficient/Insufficient/Needs Update)
- Add review comments
- Records reviewer and review date

### Test Execution Modal
Located in: `src/components/ExecuteTestModal.tsx`

**Features**:
- View test procedure
- Record test result (Pass/Fail)
- Add actual results and findings
- Upload evidence during test
- Records tester and execution date

### Upload Evidence Modal
Located in: `src/components/UploadEvidenceModal.tsx`

**Features**:
- Upload multiple files
- Set evidence type
- Add notes and description
- Set validity/expiration date
- Link to control/test/requirement

## 10. Key Features

### Traceability
- Every evidence document is linked to controls, tests, and requirements
- Complete audit trail of who uploaded, reviewed, and approved
- Version history tracks all changes

### Accountability
- Clear ownership at every level (control owner, tester, reviewer, approver)
- Timestamps for all actions
- Comments and notes for context

### Automation
- Scheduled test reminders
- Evidence expiration alerts
- Compliance score auto-calculation
- Workflow notifications

### Flexibility
- Many-to-many relationships (one evidence can support multiple controls)
- Reusable tests across controls
- Flexible tagging and categorization
